# Daogreen Spec — техническое задание

**Продукт:** лёгкий конструктор спецификаций (BOM) + калькулятор фермы + клиентский чек-лист закупки.  
**Не ERP:** без склада, закупочных ордеров, бухгалтерии, многоуровневых согласований, CRM, логина клиента.

**Суть:** проектирую ферму → собираю спецификацию из модулей → утверждаю → клиент закупает по ссылке → контролирую прогресс.

**Бренд:** Daogreen Spec · фирменный цвет `#116355` · мобильная вёрстка · PWA — в v3.

## Реализация v1.0 (актуально, июнь 2026)

| Компонент | Стек |
|-----------|------|
| **Фронт** | React 18 + Vite, SPA `/spec/`, `@tanstack/react-virtual` (материалы, закупка, спецификация, SpecPickerTable) |
| **API** | Express 4, `X-Admin-Key` (админ), `X-Client-Token` (клиент) |
| **БД** | SQLite (`backend/data/daogreen.db`), WAL, журнал миграций `schema_migrations` |
| **Бэкап БД** | Supabase Storage (опц.): `daogreen.db` + архивы `backups/daogreen-*.db` |
| **Файлы** | `STORAGE_DRIVER=local` (по умолчанию) или `s3` — Yandex/AWS/MinIO через `@aws-sdk/client-s3` |
| **Клиент** | `/client/p/:token` — закупка, статусы (вкл. «Заказано»), bulk-patch, Excel/PDF |
| **Shared** | `shared/` — merge, крепёж, stellage, readiness (front + back) |
| **Безопасность** | Helmet (без HSTS/upgrade на HTTP), rate-limit, CORS allowlist, upload filter |

**Маршруты клиента:** `GET/PATCH /api/client/p/:token/...` + заголовок `X-Client-Token`.

**Каталог на клиенте:** `GET /api/client/p/:token` → `catalog: { materials, modules, stellageGroups }` (только нужное для закупки).

**Деплой:** VPS `62.233.35.206`, systemd `daogreen-spec`, скрипты `scripts/vps-upload-dist.mjs`, `vps-deploy-step12.mjs`.

**Тесты:** `npm test` — Vitest (`tests/`, 11+ тестов на `shared/`).

**Миграции:** `backend/migrations/*.sql` + `migrateDb()` в `db.js`.

## 1. Роли доступа

| Роль | Доступ | Аутентификация |
|------|--------|----------------|
| **Администратор** (ты) | Все проекты, база материалов, шаблоны, импорт/экспорт, утверждение, архив | Простой вход (пароль / basic auth) — не в MVP-каркасе |
| **Клиент** | Только свой проект: закупка, статусы, факт. цена, комментарии, Excel/PDF | Уникальная ссылка с токеном, без регистрации |
| **Подрядчик** (опционально, v2+) | Списки по специальности: сантехник / электрик / монтажник | Отдельная ссылка или фильтр в клиентском экране |

## 2. Что НЕ делаем в первой версии

- Склад, закупочные ордера, заявки поставщикам
- Многоуровневые согласования, бухгалтерия
- Автопарсинг цен с сайтов, сложная CRM
- Личные кабинеты юрлиц, регистрация клиента по логину
- Сложная система прав (достаточно: админ + ссылка клиента)
- SKU, артикулы, штрихкоды, вес, объём, страна, бренд

### 4.1. Material (справочник материалов)

**Сейчас в коде** (`seedMaterials.js`):

| Поле | Тип | MVP | Примечание |
|------|-----|-----|------------|
| id | uuid | ✅ | |

| category | enum/text | ✅ | |
| subcategory | text | ⬜ | |
| itemType | enum | ⬜ | material / equipment / consumable / work / delivery |
| module | fk → Module.name | ✅ | |
| unit | text | ✅ | |
| defaultQty | decimal | ✅ | на 1 стеллаж или на модуль |
| basePrice | decimal | ✅ | |

### 4.2. Module (шаблон спецификации)

**Сейчас** (`modules.js`):

```ts
{ id, name, type: "stellage" | "general", tech }
```

**Целевой список модулей:**

| Группа | Модули |
|--------|--------|
| Стеллажи | подтопление, проточка, аэропоника, клубника, рассадное подтопление |
| Ферма | общая закупка, водоподготовка, охлаждение, отопление, вентиляция, влажность, освещение, электрика, полив, насосы, ёмкости, автоматы, кабель, расходники, монтажные работы, запуск |

**Расчёт:** для `type = stellage` → `qty = defaultQty × count_стеллажей`. Для `general` → `qty = defaultQty` (масштаб вручную или отдельными калькуляторами).

### 4.3. Project (проект клиента)

**Сейчас:**

```ts
{
  id, name, client, city, area, height, type,
  currency, vat, comment,
  selectedModules: [{ moduleId, count }],
  items: Item[],
  createdAt
}
```

**Добавить в MVP+:**

| Поле | Назначение |
|------|------------|
| sowingArea | посевная площадь |
| status | draft / active / archived |
| clientToken | защищённая ссылка |
| zones[] | зоны выращивания (см. §6) |
| stellageConfigs[] | параметры стеллажей (см. §6) |
| manualParams | длины труб/кабеля, расстояния и т.д. |
| version | текущая версия спецификации |
| lastClientActivityAt | для дашборда «клиент давно молчит» |

### 4.4. Item (позиция в проекте)

Позиция **не меняет** базу материалов — только переопределения в проекте.

**Сейчас:**

```ts
{
  id, materialId, module, name, unit, category,
  link, comment,
  qty, price,
  visible, approved,           // админ
  status, actualPrice, clientComment  // клиент
}
```

**Добавить:** `supplier`, `vat`, `photo`, `section` (раздел уровня 1), `enabled` (qty=0 → выключена, не удалять), `needsApproval`, `adminStatus`.

### 4.5. SpecVersion (версии, v2)

```ts
{
  id, projectId, versionNumber, createdAt, createdBy,
  summary: { added, removed, changed, sumBefore, sumAfter, delta },
  snapshot: Item[]   // снимок items на момент утверждения
}
```

Клиенту показывать: *«Спецификация обновлена. Добавлено 3, удалено 1, сумма +12 400 ₽»*.

### 4.6. File (файлы, MVP+)

```ts
{ id, projectId?, itemId?, type: photo|receipt|excel|pdf, url, uploadedAt }
```

## 6. Логика расчёта

### Автоматически

| Что | Формула / правило |
|-----|-------------------|
| Материалы стеллажа | `defaultQty × кол-во стеллажей` |
| Сумма строки | `qty × price` |
| Бюджет проекта | Σ строк, где `approved && visible && enabled` |
| Потрачено | Σ факт. суммы по статусам `bought`, `delivered`, `have` |
| Прогресс | `doneCount / totalVisible × 100%` |
| Общий список | группировка по `name + unit`, сумма qty, расшифровка `sources[]` по модулям |
| Списки специалистов | фильтр по `category → SPECIALIST_MAP` |

### Вручную (не «думать за проектировщика»)

- Длина магистрали, до канализации / воды / щита
- Длина кабеля, труб, фитинги по месту
- Вытяжка, кондиционер, отопление, вентиляция
- Зоны: насос полива, насос дренажа, рассадное отделение

### Формула проекта — входные данные

**Общие:** название, клиент, город, площадь, высота, посевная площадь, тип фермы, валюта, НДС, комментарий.

**Стеллаж (на тип):** кол-во, длина, ширина, высота, ярусы, поддоны/ярус, светильники/ярус, зона, культура, комментарий.

**Зона:** название, технология, кол-во стеллажей, насосы, рассадное да/нет, ручные длины, комментарий.

## 7. Три уровня спецификации

```
Уровень 1 — Разделы (модуль / section)
    └── Уровень 2 — Позиции (items внутри раздела)
```

Пример уровня 3: *Кран шаровый д25 — 12 шт* → 4 проточка + 3 рассадное + 5 магистраль.

## 11. Ключевые бизнес-правила (заложить сразу)

1. **Одна позиция — несколько модулей.** Один материал в базе; в проекте — отдельные `Item` на модуль (или объединение только на уровне 3).
2. **Переопределение только в проекте.** База не трогается при правке qty/price/link в проекте.
3. **Клиент видит только `approved && visible`.**
4. **Общий список объединяет одинаковые** с расшифровкой источников.
5. **Нулевые позиции не удаляются** — `enabled = false` или `qty = 0`.
6. **Скрытые строки Excel** → `visible = false`, не выкидывать из данных.

*Все 6 правил уже реализованы в MVP-каркасе* (`StoreContext`, `helpers.js`, `ClientProjectPage`).
